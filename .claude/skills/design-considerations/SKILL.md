# 横断的考慮事項チェックリスト(経験者の暗黙知)

設計時に**全カテゴリを走査**し、design.md の「横断的考慮事項」表に該当/非該当を明記する。**非該当の判断にも理由を書く**(書いていない = 考えていない、とみなす)。

## 1. データ整合性

- この操作のトランザクション境界はどこか。途中で失敗したら何が中途半端に残るか
- 同じ操作が2回実行されたら(二重送信・リトライ・再実行)何が起きるか。冪等性は必要か
- 2人が同時に同じデータを触ったら(競合)どちらが勝つべきか。楽観/悲観ロックの要否
- 一意であるべきものは何か。**DB 制約で**守られているか(アプリ層のチェックだけでは同時実行で破れる)
- 削除は物理か論理か。関連データはカスケードするか、孤児レコードになるか

## 2. 互換性・移行

- 保存済みの既存データは、この変更後も正しく読めるか
- API の変更は既存クライアント(旧バージョン・他チームの利用者)を壊さないか
- DB マイグレーションは後方互換か。**デプロイ中に新旧コードが共存する瞬間**に壊れないか
- 切り戻し(ロールバック)したとき、新形式で書かれたデータはどうなるか

## 3. セキュリティ・権限

- この機能に触れていいのは誰か。認可チェックはどの層で行うか(URL 直叩きされた場合も効くか)
- 他人のリソース ID を指定されたら(IDOR)弾けるか
- 入力はどこで検証するか。インジェクション(SQL/XSS/コマンド)の経路はないか
- 個人情報・秘密情報がログ・エラーメッセージ・URL に漏れないか

## 4. 性能・スケール

- データ量が今の100倍になっても動くか。一覧系にページネーションはあるか
- ループ内でクエリ/外部呼び出しをしていないか(N+1)
- キャッシュするなら無効化のタイミングは何か(更新時に古いキャッシュが残らないか)
- タイムアウトは設定されているか。遅い外部依存が全体を巻き込まないか

## 5. 運用・可観測性

- この機能が本番で壊れたとき、**誰がどうやって気付くか**(ログ・メトリクス・アラート)
- 失敗時のログに原因究明に足る情報があるか(ただし秘密情報は含めない)
- 段階的リリース(機能フラグ・カナリア)は必要か。問題発生時に即座に止められるか
- 手動の運用作業(再実行・データ修正)が必要になるケースはあるか。その手順は書いたか

## 6. UX・エッジケース

- 空状態(0件)・初回利用・ローディング中・エラー時の表示は定義されているか
- 二重クリック・連打で何が起きるか
- タイムゾーン・ロケール・文字数制限(絵文字・サロゲートペア)の影響はあるか

## 7. 外部依存

- 依存する外部 API・サービスが落ちていたら、この機能はどう振る舞うか(全滅か縮退か)
- リトライするなら回数・間隔は。相手から見て冪等か。レート制限に当たらないか
- 外部仕様の変更(レスポンス形式など)にどこまで耐えるか

## 運用ルール

- **該当する項目** → 対応を設計本文(アーキテクチャ・実装ステップ・テスト戦略)に必ず反映する
- **非該当の項目** → 「なぜ非該当か」を一言添える(例: 「読み取り専用機能のため競合なし」)
- このコードベース特有の罠・考慮事項を発見したら、**このファイルに追記**してチームで共有する(経験の属人化を防ぐ)
