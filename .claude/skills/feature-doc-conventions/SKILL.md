# 機能ドキュメントの構成規約

`docs/features/<slug>/` には進行中の `/feature` ワークフローの成果物が入る。**実装完了後は `docs/features/_archive/<実装完了日>-<slug>/` に移される**(削除せず履歴として残す)。

## requirements.md(要件定義書)

必須セクション: 背景・目的 / スコープ(やること・**やらないこと**) / ユーザーストーリー / 受け入れ基準 / 非機能要件 / 制約 / 未決事項 / 議論ログ。

- 受け入れ基準は「<条件> のとき <期待結果>」の検証可能な形式
- ステータス行: `draft` → 承認後 `approved`
- 該当なしのセクションは「なし」と明記(書き忘れと区別)
- 未決事項は空欄にしない。決まっていないことを正直に書く

## design.md(設計書)

必須セクション: 概要 / 現状分析 / **影響範囲分析** / アーキテクチャ / 変更ファイル一覧 / データ・インターフェース設計 / **横断的考慮事項** / 実装ステップ / テスト戦略 / **テストケース一覧** / 動作確認手順 / リスク・代替案 / **検証が必要な仮定**。

設計承認後に追記されるセクション(早期フェーズでは未記入が正常): スパイク実証結果(2-d 後) / 動作確認結果(Phase 4)。

- ステータス行: `draft` → 承認後 `approved` → 実装完了後 `implemented`
- 実装ステップは実装者の裁量が残らない粒度(シグネチャ・エラー処理・対応テストケース番号・確認方法)
- **横断的考慮事項**は全カテゴリ(7種)を埋め、非該当にも理由を書く(空欄=考えていない、とみなす)
- **テストケース一覧**は受け入れ基準と「該当」考慮事項を全カバー
- 受け入れ基準のそれぞれが設計のどこで満たされるか追跡できること

完全なテンプレートは `.claude/skills/feature/requirements-template.md` / `design-template.md` を参照。
