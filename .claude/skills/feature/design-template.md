# 設計書テンプレート

`docs/features/<slug>/design.md` は以下の構成で書く。

```markdown
# 設計: <機能名>

- ステータス: draft | approved | implemented
- 対応する要件: ./requirements.md

## 概要

設計の方針を3〜5行で。

## 現状分析

調査で分かった既存コードの構造、再利用できるもの。

## 影響範囲分析

変更が波及する箇所を**実コードで確認して**書く(推測で書かない)。

- **変更するモジュール/API の呼び出し元**: 全列挙(grep で確認したこと)
- **共有している状態**(DB テーブル・キャッシュ・グローバル設定)**とその他の利用者**: ...
- **壊しうる既存機能とその確認方法**: ...

## アーキテクチャ

データフロー・コンポーネント間の関係。必要なら mermaid 図。

## 変更ファイル一覧

| パス | 新規/変更 | 内容 |
| :-- | :-- | :-- |
| src/... | 新規 | ... |

## データ・インターフェース設計

スキーマ変更、API シグネチャ、型定義など。

## 横断的考慮事項

`.claude/skills/design-considerations/SKILL.md` の**全カテゴリを走査**した結果。非該当にも理由を書く(空欄・省略は不可)。

| カテゴリ | 該当 | 対応 / 非該当の理由 |
| :-- | :-- | :-- |
| データ整合性 | ✅/— | ... |
| 互換性・移行 | ✅/— | ... |
| セキュリティ・権限 | ✅/— | ... |
| 性能・スケール | ✅/— | ... |
| 運用・可観測性 | ✅/— | ... |
| UX・エッジケース | ✅/— | ... |
| 外部依存 | ✅/— | ... |

## 実装ステップ

コミット可能な小さい単位に分割。**実装者の裁量が残らない粒度**で書く(誰が実装しても同じものになることがゴール)。

1. <ステップ>
   - 変更点・主要シグネチャ: <関数名(引数): 戻り値 / 型定義>
   - エラー処理: <どの失敗をどう扱うか>
   - 対応テストケース: <テストケース一覧の #>
   - 確認方法: <テスト/コマンド>
2. ...

## テスト戦略

| 対象 | レベル | 観点 |
| :-- | :-- | :-- |
| ... | unit / integration / E2E | ... |

## テストケース一覧

**実装前に**列挙する(Phase 3 ではこの一覧を先にテストコード化してから実装する)。受け入れ基準と横断的考慮事項の「該当」項目をすべてカバーすること。

| # | Given(前提) | When(操作) | Then(期待結果) | レベル |
| :-- | :-- | :-- | :-- | :-- |
| 1 | ... | ... | ... | unit |

## 動作確認手順

Phase 4 でそのまま実行できる具体的なコマンド・操作手順。

## リスク・代替案

検討して採用しなかった案とその理由。想定されるリスクと対策。

## 検証が必要な仮定

実装してみないと真偽が分からない仮定を正直に列挙する(/spike-probe がここを読む)。なければ「なし」。

- [ ] <仮定> — 外れた場合の影響: <...>

## スパイク実証結果

(/spike-probe 実行後に記入)

| 仮定 | 判定 | 証拠・学び |
| :-- | :-- | :-- |
| ... | confirmed / refuted / partial | ... |

## 動作確認結果

(Phase 4 で記入)

- 実行日: <YYYY-MM-DD>
- 実行したこと: ...
- 受け入れ基準の照合:
  - [x] <基準1> — <確認方法と結果>
```
