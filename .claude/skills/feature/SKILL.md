# 機能開発ワークフロー — Feature Monte Carlo

人間のフローの模倣に加えて、Phase 1 で**解釈モンテカルロ**(曖昧さの実測)、Phase 2 で**スパイク実証**(実装証拠による設計検証)という AI にしかできない検証を行う。

依頼内容: $ARGUMENTS

## 既存の機能ドキュメント

!`ls docs/features/ 2>/dev/null || echo "(まだない)"`

以下の5フェーズを**順番に**進める。各フェーズ末尾の承認ゲートでは AskUserQuestion でユーザーの明示的な承認を取り、**承認なしに次フェーズへ進んではならない**。

## Phase 0: 準備

1. 依頼内容から機能のスラッグ(kebab-case、例: `csv-export`)を決め、ユーザーに確認する。
2. 機能の**規模**をヒアリング内容から見積もる(影響モジュール数の見込み、認可・課金など重要領域に触れるか)。これが後続の検証ステップの省略可否を決める。Phase 1 時点で変更ファイル数は確定できないので「見込み」で判断する。
3. `docs/features/<slug>/` が既に存在する場合は中断・再開のケース。下表の基準(`feature-status` と同一)で再開フェーズを判定し、ユーザーに再開位置を報告してから該当フェーズへ進む:

   | requirements.md | design.md | 再開フェーズ |
   | :-- | :-- | :-- |
   | ない or `draft` | — | Phase 1 |
   | `approved` | ない or `draft` | Phase 2 |
   | `approved` | `approved` | Phase 3(動作確認結果が未記入なら) |
   | `approved` | `implemented` or 動作確認結果あり | Phase 4〜5 |

   **サブフェーズの中断に注意**: ステータスが `approved` でも、requirements.md に解釈モンテカルロの実施記録がない/design.md の「スパイク実証結果」が空のままなら、その検証ステップ(1-d / 2-d)は未実施。承認済みでも未実施の検証があればユーザーに知らせ、実施するか確認する(サイレントにスキップしない)。

## Phase 1: 要件定義(最重要フェーズ)

### 1-a. ヒアリング

依頼内容を読み、不明点を AskUserQuestion でヒアリングする。最低限以下を埋める:

- **目的・背景**: なぜこの機能が必要か。誰のどんな課題を解くか
- **スコープ**: 何をやり、何をやらないか(やらないことを明示するのが重要)
- **受け入れ基準**: 「完成」と言える具体的な条件
- **非機能要件**: 性能・セキュリティ・互換性の制約があるか
- **運用・リリース**: 本番で失敗したときどう気付きたいか。段階的リリース(フラグ・カナリア)の要否
- **制約**: 期限、既存仕様との整合、使ってはいけない技術など

一度に全部聞かず、依頼内容から推測できるものは推測を提示して確認する形にする(「〜と理解しましたが合っていますか?」)。

### 1-b. ドラフト作成

`${CLAUDE_SKILL_DIR}/requirements-template.md` を読み、その構成に従って `docs/features/<slug>/requirements.md` にドラフトを書く。未決事項は「未決事項」セクションに正直に残す。

### 1-c. critic との議論ループ

1. **requirements-critic** サブエージェントを起動し、ドラフトのパスを渡してレビューさせる。
2. 返ってきた指摘を重大度別に整理し、**そのままユーザーに見せる**。
   - 自分で勝手に解決できる指摘 → 解決案を添えて提示
   - ユーザーの判断が必要な指摘 → AskUserQuestion で確認
3. 回答を反映してドラフトを更新する。
4. Critical の指摘が残っている間は 1 に戻る(**最大3周**。3周しても収束しない場合は残課題を「未決事項」に明記してユーザーに判断を仰ぐ)。

**エスカレーション**: 規模が大きい機能(変更ファイル10超が見込まれる、複数モジュールを跨ぐ、認可・課金など重要領域に触れる)の場合は、critic ループに加えて多視点パネルレビューの実施をユーザーに提案する:

```
/req-panel docs/features/<slug>/requirements.md
```

(PM・アーキテクト・QA・セキュリティの4視点で並列レビューする Dynamic Workflow。トークン消費が大きいので提案して承認を得てから)

### 1-d. 解釈モンテカルロ(曖昧さの実測)

critic ループ収束後、ドラフトの曖昧さを**実測**する:

```
/interpretation-mc docs/features/<slug>/requirements.md
```

互いを知らない5体のエージェントが独立に要件を解釈し、解釈が**分裂した箇所 = 曖昧箇所**として報告される。結果の扱い:

- **分裂した解釈** → AskUserQuestion でどちらが正かユーザーに確認し、要件書に反映
- **全員が補完した暗黙の仮定** → 事実上の仕様なので要件書に明文化
- Phase 0 で見積もった規模が**ごく小さい**(影響モジュール1つ程度、重要領域に触れない)場合は省略可。省略する場合はその旨ユーザーに確認する
- critic ループが非収束で Critical を未決事項に送った場合は、**先にユーザーに未決事項を埋めてもらってから**実行する(でないと解釈の分裂が要件の曖昧さでなく未決事項由来のノイズになる)

### 1-e. 承認ゲート ▶

最終版の要件定義書の要約を提示し、承認を得る。**承認後、requirements.md の冒頭ステータス行を `draft` → `approved` に更新する**(この更新を忘れると再開判定・進捗一覧が壊れる)。

## Phase 2: 設計

### 2-a. 現状調査と影響範囲分析

Explore サブエージェントで以下を調査する(新規プロジェクトの場合はディレクトリ構成方針を決める):

1. **現状**: 既存の類似機能、再利用できるもの、テストの書き方の慣習
2. **影響範囲**(推測でなく実コードで確認させる):
   - 変更対象のモジュール/API の**呼び出し元の全列挙**
   - 変更対象が**共有している状態**(DB テーブル・キャッシュ・設定)と、その他の利用者
   - この変更で**壊しうる既存機能**と、壊れていないことの確認方法

結果は design.md の「現状分析」「影響範囲分析」セクションに書く。**新規プロジェクトや既存コードに接点のない機能**では影響範囲が空になるのが正常。その場合は「新規追加のみ・既存への波及なし(確認方法: 該当ディレクトリにコードが存在しないこと)」と明記する(空欄にはしない)。

### 2-b. 設計書作成

`${CLAUDE_SKILL_DIR}/design-template.md` を読み、その構成に従って `docs/features/<slug>/design.md` を書く。重要なのは:

- **変更ファイル一覧**: 新規/変更するファイルを具体的なパスで列挙
- **実装ステップ**: **実装者の裁量が残らない粒度**で書く。各ステップに主要シグネチャ・エラー処理方針・対応テストケース番号・確認方法を付ける(誰が実装しても同じものになることがゴール)
- **テストケース一覧**: Given/When/Then で実装前に列挙。受け入れ基準と横断的考慮事項の「該当」項目を全カバー
- **動作確認手順**: Phase 4 で実際に実行するコマンド・操作を具体的に
- **検証が必要な仮定**: 実装してみないと真偽が分からない仮定を正直に列挙(2-d のスパイク対象)

設計上の選択肢が複数あり優劣が自明でない場合は、AskUserQuestion でトレードオフを提示して選んでもらう。

### 2-b'. 横断的考慮事項の走査(経験者の暗黙知)

`.claude/skills/design-considerations/SKILL.md` のチェックリストを**全カテゴリ走査**する:

- 該当する項目 → 対応をアーキテクチャ・実装ステップ・テストケース一覧に反映
- 非該当の項目 → 「なぜ非該当か」を一言添える
- 結果を design.md の「横断的考慮事項」表に記録(**全カテゴリ埋める。空欄 = 考えていない、とみなされる**)
- 走査中にこのコードベース特有の罠を発見したら、チェックリスト本体に追記してチームに還元する

### 2-c. 設計レビュー

**design-reviewer** サブエージェントを起動し、`requirements.md` と `design.md` のパスを渡してレビューさせる(要件の取りこぼし・実現性・既存コードとの整合を見る)。指摘を反映する。

### 2-d. スパイク実証(実装証拠による検証)

設計書の「検証が必要な仮定」に項目がある場合:

```
/spike-probe docs/features/<slug>/design.md
```

リスキーな仮定ごとに worktree 隔離のエージェントが**使い捨てコードを実際に書いて実行し**、証拠だけが戻る(コードは破棄される)。結果の扱い:

- **refuted(反証)** → 設計を修正し、2-c の設計レビューからやり直す。修正後に新たな検証が必要な仮定が生じたら再度スパイク。**2-c↔2-d の往復は最大2周**。収束しなければ残リスクを design.md の「リスク・代替案」に明記してユーザー判断を仰ぐ
- **partial(条件付き)** → 条件を設計書のリスク・実装ステップに反映
- 学び(API の癖、性能実測値など)を design.md の現状分析に追記し、結果をスパイク実証結果セクションに記録
- **git リポジトリでない場合**: スパイクは worktree 隔離が前提なので実行できない。検証が必要な仮定を design.md のリスクに残したまま「未検証の仮定あり」と明示してユーザーに承認を仰ぐ(または `git init` を提案)
- 仮定が1件もなければ省略してよい

### 2-e. 承認ゲート ▶

設計の要約(特に変更ファイル一覧と実装ステップ)を提示し、承認を得る。**承認後、design.md のステータス行を `draft` → `approved` に更新する。**

## Phase 3: 実装

設計書が仕様。**ここで新しい判断はしない**(判断が必要になったら設計に戻る)。

1. 各ステップの「対応テストケース」を design.md のテストケース一覧から**先にテストコード化し、失敗することを確認**してから実装する(テストが仕様の写経になるのを防ぐ)。
2. 設計書の実装ステップに順番に従い、ステップごとに進捗を報告。シグネチャ・エラー処理は設計書の記載どおりに実装する。
3. 実装中に設計と乖離する判断が必要になったら、**黙って進めず** design.md(必要なら横断的考慮事項表も)を更新してユーザーに差分を報告する。
4. 全ステップ完了後、`git diff` の要約と、テストケース一覧の消化状況(#ごとに実装済みか)を提示。

## Phase 4: 動作確認

1. design.md の「動作確認手順」を**実際に実行**する(テストを流すだけでなく、可能ならアプリを起動して挙動を観察する)。
2. 結果を `docs/features/<slug>/design.md` の動作確認セクションに記録する(実行したコマンド、結果、確認できた受け入れ基準)。
3. 受け入れ基準を1つずつ照合し、✅/❌ で報告する。❌ があれば修正に戻る:
   - ❌ の原因が**実装バグ** → Phase 3 に戻る
   - ❌ の原因が**設計の誤り**(設計どおり作ったが基準を満たせない) → **先に design.md(必要なら requirements.md)を更新し、2-c の設計レビューから**やり直す(実装だけ直してドキュメントと乖離させない)
4. **承認ゲート ▶**: 確認結果を提示し、コミットへ進む承認を得る。**承認後、design.md のステータス行を `approved` → `implemented` に更新する。**

## Phase 5: コミット

1. **code-reviewer** サブエージェントで最終レビューを実施し、Critical 指摘があれば修正する。
2. `/commit` スキルでコミットする(`docs/features/<slug>/` の成果物も含めること)。
3. 完了報告: 実装内容の要約、成果物の場所、残った未決事項・フォローアップを列挙する。

## 全体の注意

- 各フェーズの成果物は次フェーズの入力。フェーズを跨ぐ判断材料は必ずファイルに書き、会話内だけに置かない。
- ユーザーが途中で要件を変えたら、requirements.md を更新してから先へ進む(ドキュメントと実装の乖離を作らない)。
