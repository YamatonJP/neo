# neo — Feature Monte Carlo: 機能開発ワークフロー テンプレート

機能の依頼を受けてから、**要件定義 → 設計 → 実装 → 動作確認 → コミット**までを、承認ゲート付きで一気通貫に進める Claude Code 用のチーム共有テンプレート。

人間の開発フローをなぞるだけではなく、要件と設計の各段階に **AI にしかできない検証**(解釈モンテカルロ・スパイク実証)を組み込んでいるのが特徴 → [このテンプレートのアイデア](#このテンプレートのアイデア)。

## 使い方

```text
/feature ユーザー一覧に CSV 出力機能を追加したい
```

これだけで以下が走る:

```
[Phase 1: 要件定義] ★最重要
  ヒアリング → requirements-critic と議論ループ(最大3周)
  大規模機能なら /req-panel で多視点パネルレビューにエスカレーション
  → 解釈モンテカルロ: 独立した5体に要件を解釈させ、分裂箇所=曖昧箇所を実測
  → docs/features/<slug>/requirements.md
  ── ユーザー承認 ──▶
[Phase 2: 設計]
  コードベース調査+影響範囲分析(呼び出し元・共有状態の全列挙)
  → 横断的考慮事項チェックリスト(経験者の暗黙知)を全カテゴリ走査
  → テストケース一覧まで含む「実装者の裁量が残らない」設計書を作成
  → design-reviewer がレビュー
  → スパイク実証: リスキーな仮定を worktree の使い捨て実装で並列検証
  → docs/features/<slug>/design.md
  ── ユーザー承認 ──▶
[Phase 3: 実装]
  設計の実装ステップに従い、テストと同時に実装
[Phase 4: 動作確認]
  設計書の手順を実際に実行 → 受け入れ基準を ✅/❌ で照合
  ── ユーザー承認 ──▶
[Phase 5: コミット]
  code-reviewer 最終レビュー → 成果物を _archive/<完了日>-<slug>/ に移動
  → /commit(pre-commit hook でテスト強制)
```

補助コマンド:

| コマンド | 用途 |
| :-- | :-- |
| `/feature-status` | 全機能の進行状況を一覧(fork + Explore 実行で本体コンテキストを汚さない) |
| `/interpretation-mc <requirements.mdのパス>` | 解釈モンテカルロ単体実行。既存の仕様書・Issue の曖昧さ計測にも使える |
| `/spike-probe <design.mdのパス>` | スパイク実証単体実行。設計メモの仮定検証にも使える |
| `/req-panel <requirements.mdのパス>` | PM・アーキテクト・QA・セキュリティの4視点並列レビュー(トークン消費大、大規模機能のみ) |
| `/commit` | 規約に沿ったコミット(単体でも使える) |

中断しても、同じ `/feature` で `docs/features/<slug>/` の状態から再開できる。

## このテンプレートのアイデア

人間の開発フロー(要件定義→設計→実装)をそのまま AI に写すと、人間のレビューが持つ弱点(推測・思い込み・1案しか作れない)まで一緒に写してしまう。このテンプレートは、フローの骨格は人間のものを保ちつつ、**人間のチームには物理的に不可能な検証**を2箇所に埋め込んでいる。

### 解釈モンテカルロ — 曖昧さは推測せず、実測する(Phase 1)

要件の曖昧さは普通、レビュアーが「ここは曖昧そう」と*推測*する。このテンプレートでは、**互いを知らない5体のエージェントに同じ要件書を独立に解釈させ、「自分ならこう作る」を書かせて突き合わせる**:

- 解釈が**一致**した部分 = 明確な要件
- 解釈が**分裂**した部分 = 実装者によって違うものが出来上がる箇所。承認前にユーザーに確認して潰す
- 全員が**同じ暗黙の仮定**で補完した部分 = 要件書に書かれていない事実上の仕様。明文化する

曖昧さが意見ではなく**測定値**(5人中3人がA、2人がB)として出るのがポイント。人間で同じことをすると5人×数時間かかるが、AI なら毎回の機能開発で数分でできる。

### スパイク実証 — 設計レビューを意見から証拠に変える(Phase 2)

「このライブラリでできるはず」「この移行は互換性が保てるはず」という設計上の仮定は、実装が始まってから外れると手戻りが最大になる。このテンプレートでは、設計書に**「検証が必要な仮定」を正直に書かせ**、その各仮定を **worktree 隔離のエージェントが使い捨てコードを実際に書いて・実行して検証する**:

- 検証コードは worktree ごと破棄され、**証拠(confirmed / refuted / partial)と学びだけ**が設計レビューに戻る
- 反証された仮定は実装前に設計を修正(手戻りの最大要因を先に潰す)
- 人間は「捨てる前提のコード」を3本並列では書けないが、AI には使い捨てこそ得意分野

どちらも `/feature` に組み込み済みだが、単体コマンドとしても使える(既存の Issue の曖昧さ計測、設計メモの仮定検証など)。

### 暗黙知の強制走査 — 「書いていない」と「考えた上で非該当」を区別する(Phase 2)

トランザクション境界・デプロイ中の新旧共存・キャッシュ無効化・本番で壊れたとき誰が気付くか——この種の考慮は経験者の頭の中にだけあり、レビューで指摘されるかは**運**になりがち。このテンプレートでは経験者の暗黙知を7カテゴリのチェックリスト(`design-considerations`)に明文化し、設計書に**全カテゴリの該当/非該当と理由の記入を強制**する:

- 空欄 = 「考えていない」とみなされ、design-reviewer が差し戻す
- **「非該当」も疑われる**(誤った非該当は Critical 指摘)
- コードベース特有の罠を見つけたらチェックリスト本体に追記 → 経験が属人化せずチームに蓄積

さらに設計書は**「実装者の裁量が残らない粒度」**(各ステップにシグネチャ・エラー処理・対応テストケース番号)まで仕様化され、Phase 3 はテストケース一覧を先にテストコード化してから実装する。誰が(どのモデルが)実装しても同じものになる——バラつきの原因を実装フェーズから設計フェーズに移し、そこを多重レビューで固める構造。

## セットアップ

このテンプレートは**スタック非依存**(言語・フレームワークの固定なし)。プロジェクト固有の情報は **CLAUDE.md と pre-commit.sh の2箇所に集約**してあり、そこだけ書き換えれば動く。

### Step 1. テンプレート一式をプロジェクトのルートに配置する

このリポジトリを clone し、**`.git` フォルダを除外した**全ファイルを導入先プロジェクトのルートに配置する(`.git` を持ち込むと自分のプロジェクトの履歴がテンプレートのもので上書きされてしまう):

```bash
git clone https://github.com/YamatonJP/neo.git /tmp/neo
rm -rf /tmp/neo/.git                       # ← テンプレートの履歴を捨てる
cp -r /tmp/neo/. <あなたのプロジェクト>/    # .claude/ CLAUDE.md README.md .worktreeinclude docs/ LICENSE が入る
```

配置されるもの: `.claude/`(スキル・エージェント・hook・設定一式)、`CLAUDE.md`、`README.md`(後で Step 7 で書き換える)、`.worktreeinclude`、`docs/features/`、`LICENSE`。導入先に既存の `README.md` / `LICENSE` がある場合は上書きに注意。

### Step 2. CLAUDE.md を書き換える(最重要)

「技術スタック・コマンド」「規約」の2セクションがプレースホルダになっている。自分のプロジェクトの**事実**で埋める:

```markdown
## 技術スタック・コマンド
- スタック: TypeScript / Next.js 15 / PostgreSQL
- テスト: `npm test` / lint: `npm run lint` / ビルド: `npm run build`
- 開発サーバー起動: `npm run dev`(http://localhost:3000)

## 規約
- インデント2スペース、命名は camelCase
- エラーは Result 型で返す(throw しない)
```

ポイント:

- ここに書いたコマンドが **Phase 4(動作確認)でそのまま実行される**。アプリの起動方法・確認方法が具体的なほど動作確認の品質が上がる
- 「機能開発のワークフロー」セクションは**そのまま残す**(`/feature` を全員に使わせる起点)
- 曖昧な言葉を避ける(「きれいに書く」ではなく「インデント2スペース」)。200行以内厳守

### Step 3. pre-commit.sh のチェックを書き換える

`.claude/hooks/pre-commit.sh` はコミットのたびに自動実行され、失敗すると**コミットがブロックされる**。npm / make / cargo / go / pytest は自動検出のフォールバックがあるが、**プロジェクトのコマンドを明示的に書くのが推奨**(検出されないスタックでは素通りになる):

```bash
# --- プロジェクト固有のチェックをここに書く(例) ---
run_check "lint" npm run lint
run_check "test" npm test
```

### Step 4. permissions を作業域に合わせる

`.claude/settings.json` の `allow` に作業ディレクトリとよく使うコマンドを足すと、許可プロンプトが減って `/feature` がスムーズに流れる:

```json
"allow": [
  "Read(src/**)", "Edit(src/**)",
  "Bash(npm test *)", "Bash(npm run lint *)"
]
```

`deny` の秘密ファイル(`Read(.env)` 等)は**そのまま残す**こと。

### Step 5.(任意)MCP サーバーを足す

ブラウザのある Web アプリなら、`.mcp.json` に playwright を定義すると Phase 4 で実際に画面を操作して確認できる:

```json
{
  "mcpServers": {
    "playwright": { "type": "stdio", "command": "npx", "args": ["-y", "@playwright/mcp@latest"] }
  }
}
```

### Step 6. trust を承認して試運転する

1. プロジェクトで `claude` を起動し、workspace trust ダイアログを承認する(**hook と skill の `allowed-tools` は trust 後に初めて有効**)
2. `/doctor` でスキルが認識されているか、`/agents` で3エージェント(requirements-critic / design-reviewer / code-reviewer)が見えるか確認
3. **小さい機能で `/feature` を一周する**(例: `/feature ヘルスチェック用エンドポイントを追加したい`)。ヒアリング → critic の指摘 → 承認ゲート → 実装 → 動作確認 → コミット、の流れを体感してからチームに展開する
4. わざとテストを壊した状態でコミット(`git commit`)を試み、hook がブロックすることも確認しておく(`/commit` 経由でも素の `git commit` でも発火する)

### Step 7. この README を自分のプロジェクト用に書き換える

配置された `README.md` はこのテンプレートの説明文。**自分のプロジェクトの README に書き換える**(プロジェクトの概要・セットアップ・使い方など)。ワークフローの使い方をチームに残したい場合は、「`/feature <依頼内容>` で開発を始める」ことだけ一言書いておけば十分(詳細はテンプレート元リポジトリを参照、で足りる)。

### Step 8. コミットしてチームに共有する

`.claude/` `CLAUDE.md` `.worktreeinclude` `docs/features/` をコミットする。クローンした全員が同じワークフロー・エージェント・強制チェックを共有でき、各メンバーは初回起動時に trust を承認するだけでよい。

## 構成と設計原則

> **指針は CLAUDE.md / rules、手順は skill、強制は hook、隔離は subagent、計画の再利用は workflow**

| パス | 役割 | こういうときに使う |
| :-- | :-- | :-- |
| `CLAUDE.md` | チーム規約・スタック情報(毎セッション自動で読まれる。200行以内厳守) | ビルドコマンドが変わった、新しい規約を全員に常時守らせたい、新メンバーに伝える前提が増えた — そのたびにここを更新 |
| `.claude/skills/feature/` | ★オーケストレーター。5フェーズの手順+要件/設計テンプレート | 「〇〇機能を追加したい」という依頼が来たら、まず `/feature <依頼内容>`。中断した機能の再開も同じコマンドで |
| `.claude/skills/feature-status/` | 全機能の進捗一覧(`context: fork` + `agent: Explore` で低コスト実行) | 朝会前に進行中の機能の状態を把握したいとき、他メンバーの作業を引き継ぐとき、放置された draft を探すとき |
| `.claude/skills/feature-doc-conventions/` | 要件書・設計書の構成規約(背景知識、`/` メニューには出ない) | 直接は使わない。`docs/features/` 配下を読み書きすると自動ロードされ、レビューエージェントにもプリロードされる |
| `.claude/skills/design-considerations/` | 経験者の暗黙知チェックリスト(データ整合性・互換性・運用など7カテゴリ) | 直接は使わない。Phase 2 で全カテゴリ走査が強制される。**コードベース特有の罠を見つけたらここに追記**してチームに還元する |
| `.claude/skills/commit/` | コミット規約(`!`git status --short`` で現在の状態を自動注入) | `/feature` を通さない小さな修正(typo、設定変更など)をコミットするとき。Phase 5 でも内部的に使われる |
| `.claude/agents/requirements-critic.md` | 要件ドラフトの穴(曖昧さ・エッジケース)を攻撃する批評役 | 通常は Phase 1 で自動起動。既存の仕様書・Issue を書いた後に「requirements-critic でレビューして」と単体でも使える |
| `.claude/agents/design-reviewer.md` | 設計の実現性・要件カバレッジを実コードと照合してレビュー | 通常は Phase 2 で自動起動。実装前の設計メモを壁打ちしたいときに単体でも使える |
| `.claude/agents/code-reviewer.md` | コミット前の最終コードレビュー(sonnet でコスト最適化) | 通常は Phase 5 で自動起動。まとまったコード変更の直後に「code-reviewer でレビューして」でいつでも |
| `.claude/workflows/interpretation-mc` | 解釈モンテカルロ: 独立5体の解釈の分裂から曖昧箇所を実測 | 通常は Phase 1 で自動実行。既存の仕様書・Issue・PRD の曖昧さを測りたいときに単体でも |
| `.claude/workflows/spike-probe` | スパイク実証: リスキーな仮定を worktree の使い捨て実装で検証 | 通常は Phase 2 で自動実行。「このライブラリでできるか?」を会議で議論する代わりに単体実行しても |
| `.claude/workflows/req-panel` | PM・アーキテクト・QA・セキュリティの4視点並列レビュー | 認可・課金などの重要領域に触れる機能や、変更ファイル10超が見込まれる大規模機能の要件定義のとき(critic 1体では視点が足りない場合のエスカレーション) |
| `.claude/rules/feature-docs.md` | `docs/features/**` 編集時だけ自動ロードされる規約 | 直接は使わない。機能ドキュメントの運用ルールを増やしたいときにここへ追記(CLAUDE.md を太らせない) |
| `.claude/hooks/pre-commit.sh` | テスト失敗時にコミットをブロック(exit 2) | 直接は使わない。コミットのたびに自動実行され、壊れたコミットが CI に届く前に止める。導入時にチェックコマンドを書き換える |
| `.claude/settings.json` | 権限のベースライン + hook 登録 | 新しいコマンドを許可したいとき(`allow`)、危険な操作を確実に禁止したいとき(`deny`)、hook を追加するとき |
| `.claude/output-styles/team-dev.md` | 結論先行・パス明示の応答スタイル | 応答が冗長だと感じるメンバーが `/output-style team-dev` で任意適用 |
| `.worktreeinclude` | worktree 作成時に `.env` 等を持ち込む | 直接は使わない。worktree 隔離での並行作業時に環境変数ファイルが自動コピーされる |
| `docs/features/<slug>/` | 進行中の機能の要件定義書・設計書の置き場(コミットして共有) | 「この機能、なんでこういう仕様なんだっけ?」を調べるとき(議論ログに決定理由が残っている)。レビュー時の根拠として |
| `docs/features/_archive/<完了日>-<slug>/` | 実装完了済みの成果物(削除せず時系列で保管) | 過去に作った機能の意思決定を遡るとき。完了済みが進行中の一覧に混ざらないよう置き場で分離 |

## モデル戦略・トークン節約

| 処理 | モデル/方式 | 理由 |
| :-- | :-- | :-- |
| メインの進行・実装 | セッションのモデル | ワークフローの中核 |
| requirements-critic | `inherit` + `maxTurns: 15` | 要件議論は品質最優先。メインと同格 |
| design-reviewer | `inherit` + `maxTurns: 25` | 実コード検証を伴う |
| code-reviewer | **`sonnet`** + `maxTurns: 25` | 差分レビューは範囲が絞られるのでコスト最適化 |
| コードベース調査 | 組み込み **Explore(Haiku・読み取り専用)** | 探索は最安モデルで十分 |
| `/feature-status` | **`context: fork` + Explore** | 結果の要約だけ戻り、本体コンテキストを消費しない |
| ドキュメント規約の知識 | `user-invocable: false` スキル + `paths` ゲート | 必要なときだけロード |

**なぜ `/feature` 本体は fork しないのか**: `context: fork` は会話履歴を引き継がず要約だけ返す。承認ゲート・ヒアリングのような**対話が必要な手順は fork では成立しない**。fork は「明確なタスクで戻りが要約でいいもの」(進捗一覧・監査など)に限定するのが正しい。

**サブエージェントのメモリ**: 3体とも `memory: project`。レビュー知見が `.claude/agent-memory/<name>/` に蓄積され、git 共有でチーム全体に還元される。

## 設定方法

### 1. rules の設定

`.claude/rules/*.md` にトピック別ルールを置く(`.md` を再帰探索。`backend/` などのサブフォルダ可)。

```markdown
---
paths:
  - "src/api/**/*.ts"
---
# API 開発ルール
- 全エンドポイントで入力バリデーション必須
```

**推奨事項:**

- **原則 `paths` を付ける。** `paths` なしの rule は毎セッション無条件ロードされ、CLAUDE.md を太らせるのと同じコストになる。無条件にしたいものは CLAUDE.md に書けないか先に検討する。
- **1ファイル1トピック**にし、ファイル名で内容が分かるようにする(`testing.md`、`backend/api.md`)。レビューも差分も追いやすくなる。
- **CLAUDE.md は 200 行以内**。超えそうになったらトピック別に rules へ逃がすのがこのテンプレートの運用ルール。
- **「絶対守らせたいこと」は rules に書かない。** rules はコンテキスト(お願い)であって強制力がない。lint 必須・秘密ファイル禁止のような強制は hook / `permissions.deny` へ(このテンプレートの `pre-commit.sh` が例)。
- 複数リポジトリで同じルールを使うなら **symlink** で配れる。
- どのルールがいつロードされたかは `InstructionsLoaded` hook でログできる(デバッグ用)。`/memory` でも現在の読み込み状態を確認できる。

### 2. パスの設定

パス指定(glob)は rules / skills の `paths`、permissions の `Read()/Edit()` で同じ書式を使う。

| パターン | 意味 |
| :-- | :-- |
| `**/*.ts` | 全ディレクトリの .ts |
| `src/**/*` | src 配下すべて |
| `*.md` | **ルート直下のみ**(配下は含まない点に注意) |
| `src/**/*.{ts,tsx}` | brace 展開 |

**推奨事項:**

- **狭く始めて必要に応じて広げる。** `**/*` を安易に付けた rule は実質「無条件ロード」と同じで節約にならない。
- **permissions のパスは「deny は秘密、allow は作業域」**が基本形。このテンプレートでは:
  - `deny`: `Read(.env)` / `Read(.env.*)` / `Read(secrets/**)` — トランスクリプトは平文で残るため、読ませない事が唯一の防御
  - `allow`: 導入先で `Read(src/**)` / `Edit(src/**)` など作業域に限定して追加する
  - 評価順は **deny → ask → allow**(deny が必ず勝つので禁止事項はここへ)
- **作業ディレクトリ外**を参照したいときは `permissions.additionalDirectories`。ただしファイルアクセスのみで、そこの `.claude/` 設定は読み込まれない点に注意。
- **モノレポ**で他チームの CLAUDE.md を拾ってしまう場合は settings.json の `claudeMdExcludes` にパス/glob を列挙して除外する。
- rules の `paths` は「**Claude がそのファイルを読んだ・編集したとき**にロード」される。「このディレクトリで作業するとき」より細かい単位で効くことを前提に書く。

## セキュリティ

- `.env` / `secrets/**` の読み取りは settings.json で deny 済み(トランスクリプトは平文で残る)
- 他人のリポジトリを trust する前に `.claude/settings.json`(特に hooks)と各 SKILL.md の `allowed-tools` をレビューすること
- 組織として技術的に止めたいもの(`disableWorkflows` 等)は managed settings へ
